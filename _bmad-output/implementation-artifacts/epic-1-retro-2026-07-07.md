## 2. What Went Well

- **The walking skeleton paid off.** Standing up the full stack (ports-and-adapters backend, React/Vite front, `docker-compose` + `dynamodb-local`, table + GSIs + seed) in 1.1–1.2 meant every later story had a real end-to-end target. No story was blocked waiting for infrastructure.
- **The dual-GSI facet design (AD-4) landed cleanly.** `gsi_category` (PK=category, SK=price) for the single-category fast path and `gsi_listing` (PK="PRODUCT", SK=price) for composed filter/search were the right decomposition; sort (1.6) threaded through both via `ScanIndexForward` with almost no new surface area.
- **Adversarial code review earned its keep.** The three-lens setup (Blind Hunter · Edge Case Hunter · Acceptance Auditor) caught two client-triggerable 500s that tests had missed — the crafted-cursor 500 (1.3 High) and the >100-category `IN` 500 (1.5 Med) — and repeatedly found where fakes/mocks masked real DynamoDB behavior.
- **Severity fell over the epic.** As shared repo conventions (cursor codec, loop-to-fill, error envelope) solidified, findings shrank from High-heavy to Low. The team was demonstrably learning within the epic.

## 3. What Was Hard / What We Learned

- **Opaque cursor safety was a four-story slow burn.** The same design gap surfaced as: crafted cursor → 500 (1.3 High), cursor not bound to search term (1.4 deferred), not bound to category filter (1.5 deferred), and was finally closed in 1.6 by **binding the cursor to a query fingerprint (sort + search + sorted categories)**. Lesson: when a review defers "X isn't bound to the query," treat it as a design debt that compounds — schedule the consolidating fix rather than re-deferring per story.
- **`Limit` before `FilterExpression` — the loop-to-fill lesson.** DynamoDB applies `Limit` before filtering, so a filtered page can under-fill or come back empty while more matches exist. Predicted as a 1.3 deferral, it became the whole engineering point of 1.4 (loop-to-fill: keep following `LastEvaluatedKey` until `limit` matches accumulate). Reused by 1.5 and 1.6.
- **Test fidelity vs. real DynamoDB bit us repeatedly.** moto omits the terminal `LastEvaluatedKey` that real DynamoDB returns at an exact `Limit`, so the "empty page + phantom Load-more" bug passed tests but would misbehave in prod (recurred 1.3, 1.4 High, 1.5). Fakes also masked the crafted-cursor 500 (1.3) and a no-op sort passthrough (1.6). Lesson: for pagination/cursor behavior, assert against real-DynamoDB semantics (or an end-to-end decode test), not just moto.
- **DynamoDB has no `lower()`.** Case-insensitive search forced a stored lowercased `searchText` attribute (1.4) and an idempotent **re-seed** to backfill — which is exactly why the seed being idempotent (FR-16) mattered in practice, not just on paper.

## 5. Next Epic Preparation — Epic 2 (PDP, story 2.1)

Epic 2 is small (single story, `GET /products/{id}` + a not-found state) and governed by AD-1 and AD-5 only. It **reuses the ProductsRepository and the read path from Epic 1** — no new table, no new GSI, no cursor. Readiness is high. Carry-over that directly touches Epic 2:

- **Route ordering (Action Item 2).** Register `/products/{id}` *after* `/products/categories` in the router and add the regression test — the deferred item explicitly names this as the Epic 2 risk.
- **404 + error envelope (Action Item 1).** The PDP not-found state is the first place the frontend must render a specific error code from the AD-5 envelope. Doing Action Item 1 here (rather than deferring again) makes 2.1's not-found AC clean.
- **`_from_item` bare `KeyError`** (deferred since 1.2). PDP does a single-item `GetItem`; the same mapper runs. Still unreachable (repo owns all writes), remains an acceptable deferral — noted for awareness, not blocking.

**No significant discovery requires re-planning Epic 2.** The one architecture-scale limitation from Epic 1 — the `gsi_listing` single hot partition — does not affect the PDP (a keyed `GetItem`), so it stays deferred to the AD-4 production path (shard the listing partition or move search/facets to OpenSearch) and does not gate Epic 2.
