## 3. What Went Well

- **The Epic 3 → Epic 4 handoff was frictionless.** The atomic-transaction and Orders-table action items were written into Story 4.3 up front, so the epic's one genuinely new technical pattern (`TransactWriteItems` across the Orders + Carts tables) was planned, not discovered mid-flight.
- **Two-pass review caught two would-ship defects.** The formal `/bmad-code-review` found (a) **client-trusted shipping data** — `ShippingDetails` was bare `str`, so a direct POST could persist an immutable order with an empty address / invalid email — and (b) a **blanket `ClientError → 409`** that masked 5xx faults as retryable conflicts. Both fixed before commit. On the epic that persists the customer's order, "the server is the last line of validation" is the lesson that matters most.
- **Atomicity is provably correct.** `test_place_order_txn_writes_order_and_clears_cart` asserts the order is written **and** the cart deleted in the same transaction; the immutable snapshot round-trips. AD-7 isn't just claimed — it's tested.
- **Refactor-with-confidence.** Extracting the shared `core/guest.resolve_guest_id` (used by cart + checkout) was safe because the Epic 3 cart tests immediately confirmed no behavior change.

## 4. What Was Hard / What We Learned

- **Client validation is never the authority.** The shipping form validated inline (good UX), but the server trusted it until review. The order's shipping data is the entire deliverable — it must be validated server-side (now `min_length` + email at the model). A recurring theme all project: PLP `>100 categories`, PDP oversized id, cart quantity cap, and now shipping — *bound and validate untrusted input at the boundary, every time.*
- **A patch is new code.** Consistent with Epic 3's busy-lock regression, this epic's error-path fixes needed their own scrutiny (the `refresh()`-coupling and `empty_cart` routing). The two-pass habit is now the standing rule for state/error paths.
- **Cross-aggregate atomicity bends "one repo per table."** The order+cart-clear transaction inherently spans two tables; it lives in `OrdersRepository` (the placement owner) taking the carts table name. A deliberate, documented AD-3 nuance that AD-7 forces.
